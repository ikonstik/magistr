# Booking-сервис

#### Для каждого эндпоинта в коллекции создан пример ответа, для некоторых примеры отрицательного и положительного ответа

### 1) Авторизация пользователя ([POST] /api/auth/login)

```
На вход от пользователя передаются:
    - email
    - пароль

Тесты:
    - Проверка статус кода (200)
    - Получение токена и сохранение его в переменную
```

### 2) Получение списка мероприятий ([GET] /api/events)

```
В заголовках передается токен авторизации.
Параметры фильтрации:
    - sportType
    - date
    - location
    
Тесты:
    - Проверка статус кода (200)
    - Проверка является ли ответ JSON
    - Проверка наличия ключевых полей (id, title, date)
```

### 3) Создание брони ([POST] /api/bookings)

```
В заголовках передается токен авторизации.
На вход от пользователя передаются:
    - event_id
    - количестов билетов
    - итоговая стоимость
    
Тесты:
    - Проверка статус кода (201)
    - Проверка наличия уникального booking_id в ответе
    - Проверка статус кода (400) для негативного теста без event_id
    - Проверка, что причиной ошибки является отсутствие event_id в запросе
```

### 4) Оплата брони ([POST] /api/payments)

```
В заголовках передается токен авторизации.
На вход от пользователя передаются:
    - booking_id
    - количестов билетов
    - итоговая стоимость
    
Тесты:
    - Проверка статус кода (200)
    - Проверка изменения статуса отпаты на PAID
    - Проверка статус кода (400) для негативного теста
    - Проверка отправки сообщения об ошибке
```

### 5) Получение деталей конкретной брони по ID ([GET] /api/bookings/{bookingId})

```
В заголовках передается токен авторизации.
    
Тесты:
    - Проверка статус кода (200)
    - Проверка наличия требуемых ключей в ответе, а также валидация значений для конкретной брони
    - Проверка статус кода (404) для брони с несуществующим booking_id
    - Проверка отправки сообщения об ошибке
```
